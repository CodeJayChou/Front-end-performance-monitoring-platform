Core Orchestration 核心编排层，本质不是一个模块而是一个控制流中心。

控制流中心：什么时候做，以什么顺序做，哪些模块二参与、失败怎么兜底、数据怎么流转 

要解决的核心问题 ？ 

事件驱动执行图调

结构化直觉？
-   Orchestration = 一个负责流程的模块 
-   更深层的理解是： 系统内决策全的重新分配机制 
-   没有编排能力会导致以下问题=》
    每个模块二都有自己的判断权力 error perf decide network =>结果就是想联邦制，每个模块都是一个国家？
-   有了编排后系统像一个中央调度系统 

前端SDK 的真实问题不是数据怎么采集，请求怎么发送？而是系统什么时候开始不可预测
-   时间维度 
-   状态混乱
-   决策混乱

可复用结构化决策模式 
-   

本质是把一个高自由度混沌系统，改造成有一个低自由度可预测系统 。
思考一个问题：Orchestration 是怎么来的 ？
主动设计？真实情况是当系统复杂度超过某个阈值是必然涌现的结构 
触发它的出现不是需求，而是失控信号 

哪些现象代表了失控信号 ？
-   数据顺序开始不可预测
-   模块之间开始相互影响
-   修改一个点导致全局行为变化
-   无法解释为什么这个数据是这样子的
这时候 Orchestration 不是选项而是唯一恢复可控性的方式 

反熵结构 
内部是降熵机制 

三层模型 
-   结构层面    执行pipeline ( 管道 )
-   控制层      统一系统决策权
-   本质层      压缩系统自由度，降低熵、使系统可以预测
